# Constraints / Risks Analysis

## Constraints

### Architectural

- **Sequential query execution in a single GraphQL resolver:** The `OverviewPage` resolver (`internal/analytics/resolver.go:31`) delegates to `GetOverviewPage` which runs all chart queries sequentially. The GraphQL schema exposes `overviewPage` as a single query returning all charts at once. This architectural choice means the response time is the SUM of all chart queries rather than the MAX. This is not a hard constraint -- it can be changed to parallel execution -- but it is the current architecture that must be understood. Source: `internal/analytics/service.go` lines 24-55.

- **gqlgen code generation dependency:** The GraphQL resolvers are generated by gqlgen (v0.17.45). Any changes to `schema.graphql` require running the gqlgen code generator to update resolver interfaces. Adding new query types or modifying the schema is possible but involves a regeneration step. Source: `go.mod` and `cmd/analytics-api/main.go` line 38.

- **Monolithic frontend query pattern:** The dashboard frontend uses a single `OVERVIEW_PAGE_QUERY` that fetches all data at once (`apps/dashboard/src/api/analytics.ts`). Switching to per-chart queries requires changing the frontend data fetching strategy, which is a significant but feasible change. The `CHART_DATA_QUERY` already exists as an alternative.

### Technical

- **ClickHouse table structure is fixed (within scope):** The events table has ~500M rows, partitioned by `(tenant_id, toYYYYMM(timestamp))`, with primary key `(tenant_id, event_type, timestamp)`. The requirements state that ClickHouse infrastructure changes (schema migrations, cluster scaling) are "not included" in scope but may be acceptable "if needed." Creating materialized views is technically possible (ClickHouse supports them natively) and does not require schema migration of the base table, but it is at the boundary of the stated scope. Source: `migrations/001_analytics_tables.sql`.

- **No secondary indexes on user_id:** The SQL migration comments note (line 25): "No secondary indexes on user_id or session_id. Queries filtering by user_id require full partition scans." The `user_retention_cohort` query performs `GROUP BY user_id` in a subquery, which would scan the full partition. Source: `migrations/001_analytics_tables.sql`.

- **Recharts SVG rendering limitation:** Recharts renders individual SVG elements for each data point. This is a library-level constraint -- there is no configuration option to enable virtualization or canvas-based rendering in Recharts. For datasets with 10k+ points, this causes 2-5 seconds of rendering time. Mitigations are limited to: (a) downsampling data before passing to Recharts, (b) switching to a different charting library (canvas-based like Chart.js, ECharts, or Plotly), or (c) implementing custom virtualization. Source: `apps/dashboard/src/components/Chart.tsx` WARNING comment on line 42.

- **Apollo Client cache ineffectiveness:** The overview page passes `new Date().toISOString()` as the `to` variable (`apps/dashboard/src/pages/Overview.tsx` lines 26-27). Since this changes on every call (down to the millisecond), Apollo's cache-first strategy never produces a cache hit. The cache key is deterministic based on query + variables, so identical time ranges would hit cache, but the current code never produces identical time ranges.

### Business

- **2-second performance target (PM requirement):** Charts must load in under 2 seconds. This is a hard business constraint. It is not specified whether this is P50, P95, or P99, or whether it refers to cold (first load) or warm (cached) scenarios. Source: `requirements.draft.md`.

- **50,000 daily active users:** The system serves ~50k DAU. Performance changes must not introduce instability at this scale. Any caching solution must handle concurrent access safely (the existing cache implementation uses `sync.RWMutex`, which is appropriate).

- **Main overview page scope only:** The optimization is scoped to the main overview page. Other dashboard pages are explicitly out of scope. Source: `requirements.draft.md` Scope section.

### Compatibility

- **GraphQL API contract preservation:** The requirements flag that it's unknown whether other consumers use the same GraphQL API. The CORS configuration (`Access-Control-Allow-Origin: *` in `cmd/analytics-api/main.go`) suggests the API may be publicly accessible. Changes to the GraphQL schema response types (e.g., adding pagination fields, changing `dataPoints` structure) could break unknown consumers. The safest approach is to make backend optimizations transparent to the API contract. Source: `cmd/analytics-api/main.go` line 49, `internal/analytics/schema.graphql`.

- **Existing chart behavior and appearance:** The 8 charts on the overview page must continue to show the same data and visualizations. Users should not notice behavioral changes beyond faster loading. Source: implied by the requirements' focus on performance, not functionality.

### Regulatory/Compliance
- None identified. The code handles tenant-scoped analytics data but no PII handling, encryption requirements, or compliance annotations are visible in the codebase.

## Risks

| Risk | Category | Likelihood | Impact | Evidence | Mitigation Idea |
|------|----------|-----------|--------|----------|-----------------|
| `user_retention_cohort` query has a parameter binding bug: 5 `?` placeholders but only 3 parameters passed via `ExecuteQuery` | technical | high | medium | `queries.go` lines 66-85 has `?` in both subqueries; `client.go:ExecuteQuery` passes only `params.TenantID, params.TimeFrom, params.TimeTo` | Fix the query to use CTEs or adjust `ExecuteQuery` to accept variable parameter counts for this specific query |
| Parallelizing 8 ClickHouse queries may exhaust the connection pool | technical | medium | high | `client.go` uses `clickhouse.Open(opts)` with no explicit pool size configuration; default pool size in clickhouse-go/v2 may be insufficient for 8+ concurrent queries per request * many concurrent users | Configure explicit connection pool size in ClickHouse client initialization; test under load |
| Data downsampling introduces visible artifacts in charts | technical | medium | low | Charts currently render all data points; downsampling to 200-300 points for line/area charts may hide short spikes or dips | Use a perceptually-aware algorithm like LTTB (Largest Triangle Three Buckets) rather than naive interval sampling |
| Caching stale data during incidents or rapid changes | scope | low | medium | If cache TTL is set too high, the dashboard may show outdated metrics during a live incident where users need real-time visibility | Use short TTL (60-120 seconds); provide a manual refresh/cache-bust mechanism on the frontend |
| Breaking unknown GraphQL API consumers when changing response behavior | integration | low | high | CORS allows all origins; `CHART_DATA_QUERY` exists in the frontend suggesting the per-chart endpoint may be used elsewhere | Keep the `overviewPage` query backward-compatible; add new queries/fields instead of changing existing ones |
| Insufficient ClickHouse capacity for parallel queries under full user load | technical | medium | high | 50k DAU, if 1% are concurrent that's 500 users * 8 parallel queries = 4000 concurrent ClickHouse queries. No capacity information available. | Caching is essential to reduce ClickHouse load; monitor query throughput after enabling parallelism |
| Cache memory growth from unbounded cache entries | technical | low | medium | `cache.go` eviction strategy only removes one expired item when full; if no items are expired, it still adds the new item, exceeding `maxSize` | Fix the eviction strategy: add LRU eviction or reject writes when at capacity with no expired items |
| Frontend progressive loading introduces layout shift (CLS) | regression | medium | low | If charts load at different times and the grid layout shifts as they appear, this degrades perceived quality even if load time improves | Use fixed-height chart placeholders/skeletons that match the final chart dimensions |

## Integration Dependencies

- **ClickHouse database:** Explicit contract -- SQL queries in `queries.go` depend on the `events` table schema defined in `migrations/001_analytics_tables.sql`. The table structure is stable (no migration framework or versioning is visible). Adding materialized views would be additive and would not change the base table contract. Failure mode: if ClickHouse is slow or unavailable, the `GetOverviewPage` function will either timeout or return partial results (charts that fail are skipped). No circuit breaker or timeout configuration is visible.

- **GraphQL API (self):** The frontend depends on the GraphQL API at the endpoint configured by `VITE_API_URL`. The API contract is defined by `schema.graphql`. Changes to response types or field names would require coordinated frontend and backend changes. The single `overviewPage` query is the primary integration point.

- **No external service dependencies beyond ClickHouse:** The analytics API does not call other services, APIs, or message queues. The dependency graph is simple: Frontend -> GraphQL API -> ClickHouse.

## Backward Compatibility

| What Changes | Current Consumers | Migration Needed? | Rollback Safe? | Notes |
|-------------|-------------------|-------------------|----------------|-------|
| Backend query execution (sequential -> parallel) | GraphQL API response format unchanged | no | yes | Internal optimization, transparent to consumers |
| Adding cache layer | GraphQL API response format unchanged | no | yes | Internal optimization, can be disabled by setting TTL=0 |
| Adding row limits to queries | Dashboard frontend, unknown API consumers | no (if limits are generous) | yes | Consumers may receive fewer data points but response structure is unchanged |
| Frontend: switching from single query to per-chart queries | Only the dashboard frontend itself | no external migration | yes | Backend already supports `chartData` query; only frontend code changes |
| Adding materialized views in ClickHouse | No external consumers of raw ClickHouse queries | no | yes (views are additive) | Existing queries can be redirected to views without schema changes |

This task primarily involves internal optimizations that do not modify external-facing interfaces. The GraphQL schema, response types, and endpoint URLs can remain unchanged. The main backward compatibility concern is if data downsampling causes visible differences in chart rendering, which is a UX concern rather than an API contract concern.

## Sensitive Areas

- **`internal/analytics/service.go:GetOverviewPage`:** This is the hottest code path in the application -- every overview page load goes through it. It handles 50k DAU worth of traffic. Changes here must be carefully tested for correctness, concurrency safety (when parallelized), and error handling (the current graceful degradation pattern must be preserved). Sensitivity: **high**.

- **`internal/clickhouse/queries.go`:** Contains the raw SQL for all chart queries. Any query modification (adding limits, changing granularity, rewriting joins) directly affects the data users see. The `user_retention_cohort` query already has a likely bug. No test coverage exists. Sensitivity: **high**.

- **`internal/cache/cache.go`:** Currently unused, so low risk from existing behavior, but the eviction bug (can exceed `maxSize`) means it needs fixing before being put into production use at scale. Sensitivity: **medium**.

- **`apps/dashboard/src/pages/Overview.tsx`:** The main user-facing component. Changes to data fetching strategy (single query vs. multiple queries) affect the loading UX, error handling, and perceived performance. No frontend tests exist. Sensitivity: **medium**.

- **ClickHouse query performance under concurrent load:** Currently, queries are sequential, so ClickHouse processes one query at a time per request. Parallelizing queries means ClickHouse receives 8x more concurrent queries. For 50k DAU, this significantly changes the load profile. If ClickHouse doesn't have capacity headroom, this could cause cluster-wide performance degradation. Sensitivity: **high** (but partially mitigated if caching reduces overall query volume).

## Self-Critique Notes

- The `user_retention_cohort` parameter binding bug assessment is based on counting placeholder `?` characters vs. parameters passed. I have high confidence this is a real bug, but I cannot verify it without runtime testing. It's possible that ClickHouse's Go driver handles repeated parameters differently, or that the query was never actually called successfully (the service skips failed charts silently).

- The connection pool risk when parallelizing is real but hard to quantify without knowing the clickhouse-go default pool settings. The default may already support 8+ concurrent queries per connection, or it may need explicit configuration. This needs documentation/library investigation during planning.

- I rated the "breaking unknown API consumers" risk as low likelihood because the optimization approach (parallel execution, caching) should be transparent to the API contract. However, if the planning stage decides to change the GraphQL schema (e.g., add pagination), this risk escalates to medium-high.

- The ClickHouse capacity concern (500 concurrent users * 8 parallel queries) assumes no caching. With caching, the actual ClickHouse query volume would be much lower (only cache misses generate queries). The risk is primarily during cache cold-start or cache invalidation events.

- I could not assess deployment risks (rollback strategy, feature flags, canary deployment) because no CI/CD configuration, deployment scripts, or feature flag infrastructure was visible in the repository.
